## 5. T19 prefs-migration blueprint (distilled — full report was in a Sonnet subagent transcript)

Migrate the ~15 bespoke `apply*()` fns + hardcoded swatch containers → ONE `Prefs({prefix:"yuma-",
mount:"#settings", schema:[…]})` from `lib/prefs/prefs.js`. Field ids chosen so `"yuma-"+id`
matches each existing localStorage key. Groups mirror the drawer IA (Typography/Colour/Screen/Layout).

**Fields & hooks:**
- *Pure `attr`/`cssVar` (no custom code — `normalize()` handles the old clamps):* `bezel`
  (data-bezel), `sheen` (data-sheen — SEE WARNING), `dosspanel`, `imgcolor`, `cards`, `frame`,
  `frametint` (theme-tint is pure CSS, no JS), and the 3 doc-font fields (`docfont-title/head/body`
  → `--doc-font-*`, `preview:true`).
- *`cssVar` + `attr` + `preview:true`:* `font-head` (--font-head + data-font-head), `font-body`
  (--font-body + data-font-body). The module's built-in preview subsumes `wireFontPreview` (it
  applies var+attr on hover, restores on leave, commit-only persist).
- *Custom `apply`:* `color` (sets 5 --green* vars + composed --glow), `bg` (3 --bg* vars + the
  `<meta name=theme-color>`), `density` (toggles body.compact class), `textsize` (see cross-field),
  `crt` (toggle, toggles body.crt class — see WARNING).
- *Cross-field:* **Text Size "auto"** derives from the current body font. Handle in `textsize.apply`
  (`body.dataset.textsize = value==="auto" ? defaultSizeForFont(prefs.get("font-body")) : value`)
  AND a top-level `onChange` nudge (commit-only): `if(id==="font-body" && prefs.get("textsize")==="auto")
  body.dataset.textsize = defaultSizeForFont(value)`. Do NOT put it in font-body.apply (apply can't
  tell commit from hover-preview; onChange is commit-only by construction).

**⚠ TWO CRITICAL LANDMINES (would ship silent regressions):**
1. **`crt-screen-integration.js` breaks.** It (a) does `getElementById("crt-toggle")` to mount its
   "Enhanced CRT" button — but prefs.js's toggle has NO id → feature silently disappears. Fix its
   lookup to `querySelector('[data-field="crt"] .prefs-toggle')`. And (b) `nativeWanted()` does a raw
   `localStorage.getItem("yuma-crt") !== "0"` — prefs.js writes `"on"/"off"` (even on silent boot), so
   this becomes permanently true → scanlines force ON. Fix: also accept `"off"`. **Must fix both.**
2. **`sheen` not `glass`.** The Screen-Glass field id MUST be `"sheen"` (key has always been
   `yuma-sheen`, despite `applyGlass`/`#glass-swatches`/"Screen Glass" label). Use `id:"glass"` and
   every existing user silently loses their glass setting.

**Also:** run the legacy `yuma-font → yuma-font-head/body` migration (PRESET_MIGRATE) BEFORE
constructing `Prefs()`. Keep a small `refreshCur`-equivalent (reads `prefs.get`, wired via
`prefs.on()`) for the `#cur-*` group-summary spans — the module's `summary:true` can't reproduce the

**Substrate token vocabulary — CONFIRMED CONCRETE (was abstract before).** `@substrate/tokens`
is real/populated (Phase 1, 2026-07-03): DTCG JSON → Style-Dictionary → `dist/tokens.css`
(gitignored). Prefix is **`--suite-` today, renaming to `--sub-` (card L0, pending)** — treat the
prefix as a variable; align on the semantic/structural leaf names:
- **semantic colours** (the themeable set): `color-surface`, `color-surface-raised`, `color-text`,
  `color-text-muted`, `color-border`, `color-accent`, `color-on-accent`
- **primitives:** `color-gray-50..900`, `color-brand-500/600/700`
- **space** `0,1,2,3,4,5,6,8` (rem: 0/.25/.5/.75/1/1.25/1.5/2 — NOTE no 7); **radius** `sm/md/lg/full`;
  **font-family** `sans/mono`, **font-size** `1..6`, **font-weight** `regular/medium/bold`.
- **Theme mechanism:** primitives in `:root`; each theme re-declares ONLY the 7 semantic colour
  aliases under `:root[data-theme="…"]`. **This is the pattern a "faction" must mirror.**

**Alignment REASSESSMENT (supersedes §2's "do-regardless token-name alignment" optimism):** the
roster's phosphor language and Substrate's surface/text model **do not share a colour or spacing
vocabulary** — `--green*` is monochrome (no surface/text split); roster `--sp-*` is px 4/8/12/16/24…
and diverges from Substrate's rem `space` at step 5. A wholesale rename would invent false
correspondences AND chase a prefix that's mid-rename. **The one thing that maps 1:1 is the theming
MECHANISM** (root-attribute → semantic override set) — i.e. exactly what **T21 factions** builds.
So the integration payoff lives INSIDE T21, not in a standalone rename. Roster theming today is
imperative (`applyColor`/`applyBg` `setProperty` the `--green*`/`--bg*` vars from JS `THEMES`/`BGS`
tables; 8 colour + 5 bg presets). Converting that to declarative `:root[data-theme]` blocks aligns
with Substrate but is **decision-gated like T19** (throwaway if we port to Lit/DTCG later; also
splits the single JS source + loses the computed `--green-rgb`). Do it as part of T21, not before.

**Net:** "eye to integration in the future" = keep using tokens + keep the theme seam clean; do NOT
rewrite the theme engine or rename tokens until the reference-vs-port call (still open) lands.
