# Security & Correctness Audit — 2026-07-02

Full-file review of `app.js` (all ~1,800 lines), `index.html`, and the data flow from the
two remote sources (Google Sheet CSV, Google Docs HTML export) to every `innerHTML` sink.
Context: the site had just gone public on GitHub Pages, which changes the threat model —
escaping bugs and URL-parameter behaviour that were harmless on a local file become
reachable by anyone with a link.

**Verdict: fundamentally sound.** Text is escaped at every sink (`esc`/`escAttr`), the doc
sanitizer is a strict allowlist rebuild (no attributes survive except validated `id`,
protocol-checked `href`/`src`), `javascript:` URLs are blocked, name/slug interpolations use
a safe generated charset. The findings below are the residue.

---

## Fixed in this audit (commit refs in git log)

### 1. `?sheet=` override worked in production — content spoofing (MEDIUM, security)
`effectiveSheetId()` honoured `?sheet=<id>` and a `yuma-sheet-override` localStorage key on
ANY host. Anyone could craft `https://<site>/?sheet=<their-sheet>` and the deployed site
would render **their** spreadsheet's content — attacker-authored dossiers under this
project's own URL and branding. Escaping prevents script injection, but spoofed *content*
on a trusted origin is a real social-engineering vector (and the link looks legitimate).

**Fix:** the `?sheet=`/localStorage overrides now only apply when `location.hostname` is
localhost. The `window.YUMA_SHEET_OVERRIDE` hook (injected by `tools/preview.py` into the
*served* copy only, never the repo file) still works unconditionally — anything able to
inject a script global has already won, so gating it adds nothing.

### 2. Silent data loss when localStorage fills (HIGH for current config, correctness)
With `CONFIG.webAppUrl` unset (the current deployed state), photos, screenshots, personal
logs and icon picks save **only** to localStorage — and screenshots are stored as 1280px
PNG data URLs. Two or three of those exhaust the ~5MB quota; `setItem` then throws
`QuotaExceededError`, the user sees a generic "⚠ script error" toast (from the global
error handler), and the save they just watched appear on screen evaporates on reload.

**Fix:** all character-data writes go through `lsSetData()`, which catches the quota error
and shows an explicit "device storage is full — this won't survive a reload" toast.
*Longer-term fix is the Apps Script backend (see Deferred #D6).*

### 3. Failed Refresh blanked the working roster (correctness)
`load()`'s catch showed the full-screen error state even for `isRefresh=true`, replacing a
perfectly good on-screen roster because one background re-fetch hit a transient network
error. **Fix:** on refresh failure with a loaded model, keep the view, set the link status
to OFFLINE, and toast — the error screen is for first load only.

### 4. Google Docs "Subtitle" style rendered as body text (correctness)
`docClean` special-cased only the `title` class (the current Guide doc styles two lines as
Title). A doc using Google's actual Title + Subtitle styles would render the subtitle as a
plain paragraph. **Fix:** `subtitle`-class paragraphs now also map to `.doc-subtitle`;
first Title remains the `.doc-title` masthead.

### 5. Malformed permalink threw an uncaught error (robustness)
`openFromHash()` called `decodeURIComponent` on the raw hash — `#c=%` (easy to produce by
truncating a shared link) threw, surfacing an error toast on load. **Fix:** wrapped; a
malformed slug now just falls through to the default view.

## Deferred — reviewed, judged acceptable, with rationale

- **D1. `esc()` (no quote escaping) used inside some attributes** — e.g.
  `data-slug="${esc(ch.slug)}"`. Safe today because slugs are generated with charset
  `[a-z0-9-]` and tags pass through `escAttr`. *Convention going forward: `escAttr` for
  anything inside an attribute, even when the value looks safe.* Not worth a churn-pass now.
- **D2. DOM clobbering via doc heading ids** — a doc-injected element with an app id (e.g.
  `id="toast"`) precedes some app elements in DOM order and could shadow `$"#toast"`
  lookups. Google generates export ids (`h.xxxx`) so authors don't control them directly;
  worst case is a broken toast, not script execution. Revisit only if DOCS ever includes
  docs from untrusted editors.
- **D3. Search index includes image URLs** — `c.search` is built from the raw joined row, so
  searching "http" or "png" matches characters with photo URLs. Cosmetic.
- **D4. Sheet-provided `<img src>` accepts any https URL** — sheet editors (semi-trusted
  tribe members) can point portraits at any host, which reveals viewer IPs to that host.
  Standard for user-content sites; acceptable at this trust level.
- **D5. Passphrase UX (dormant)** — `getPass()` keeps the passphrase in sessionStorage and
  sends it in every write POST. Fine for the intended "shared tribe passphrase" model over
  HTTPS, but see D6 — the Apps Script must be the actual enforcement point.
- **D6. The Apps Script backend (`apps-script.gs`) is the next real security surface.**
  When `webAppUrl` goes live, audit it *then*: rate-limit writes, cap upload sizes
  server-side, validate `usename` targeting (a write should not create rows), constant-time
  passphrase compare, and never echo the passphrase in errors. The client-side patterns
  (unauthorized → drop cached pass → re-prompt) are already correct.
- **D7. Selection is index-based across refreshes** — if sheet row order changes between
  refreshes, `state.selected` may land on a different character. Rare and low-impact;
  fix would be slug-based selection.

## Explicitly checked, no issue found

- CSV parser: correct RFC-4180 quote/escape/newline handling; trailing `\r` in quoted cells
  is stripped later by `cleanField`'s per-line trim.
- Name-linkify regex built from sheet names: all names are regex-escaped before joining;
  zero-length-match guard present. No ReDoS shape (flat alternation).
- `docText` quote-wrapping regex is bounded (`{1,300}?`) — no backtracking blowup.
- `highlightMatch`, `emptyStateHTML`, `renderLead`, dossier/card templates: all interpolations
  escaped or safe-charset.
- `docClean` `<a>`: only `https?:` (new tab, `noopener noreferrer`) or `#` fragments survive.
  `<img>`: only `https?:`/`data:image/`; SVG-in-img cannot run script.
- Doc reader fetch has a 15s abort; failure path retries on next tab visit (docid only set
  on success).
- Modal focus trap, Escape handling, keyboard activation paths: no state desync found.
